> 작성자: 채훈
> 목적: `knowledge_pipeline.py`가 기대하는 `KnowledgeRepository` 최종 스펙 확정
> 전제: 아래 3가지 결정은 이미 확정됨
>
> 1. **카테고리 처리**: A안 — Repository가 `category_name` 문자열을 받아 내부에서 Category 테이블 upsert 후 FK 세팅
> 2. **유저 매핑**: 하드코딩 금지 — ERD의 `user_channel_identity(provider='kakao', provider_user_id=...)` 기반 upsert
> 3. **청크 테이블명**: ERD 기준 `knowledge_chunk`로 통일 (`youtube_knowledge_chunk` → `knowledge_chunk` 리네이밍)

---

## ⚠️ 용어 주의 — ERD에 `kakao_user_id` 컬럼은 없음

이 문서에서 `kakao_user_id`는 **파이프라인 내부 변수명**일 뿐입니다. ERD에는 `kakao_user_id`라는 컬럼이 없고, 카카오 해시값은 아래 컬럼에 저장됩니다.

| 파이프라인 변수 | 값 소스 | 저장되는 ERD 컬럼 |
|---|---|---|
| `kakao_user_id` | 카카오 웹훅의 `userRequest.user.id` (해시) | `user_channel_identity.provider_user_id` (이때 `user_channel_identity.provider = 'kakao'`) |

따라서 payload에 `kakao_user_id` 키가 등장할 때마다 "값을 어디에 저장하느냐"의 답은 항상 **`user_channel_identity` 테이블의 `provider_user_id` 컬럼(provider='kakao' 조건부)** 입니다.

## 1. 파이프라인이 호출하는 메서드 4종 (이 스펙 그대로 구현 요청)

`app/services/knowledge_pipeline.py`는 이미 아래 4개 메서드를 호출하도록 반영되어 있습니다. 각 메서드의 입력/출력/동작을 그대로 맞춰 주세요.

### 1-1. `create_pipeline_record(payload: dict) -> UUID`

**호출 시점**: Step 1 (수집+청킹) 종료 직후, 채훈이 호출

**입력 payload** (파이프라인 내부 키 → ERD 컬럼 매핑 주석 포함):
```python
{
    # ↓ 파이프라인 변수명. ERD 컬럼은 user_channel_identity.provider_user_id (provider='kakao')
    "kakao_user_id": str,        # 예: 카카오 웹훅 userRequest.user.id (해시)
    "video_id": str,             # 유튜브 video_id → youtube_metadata.video_id
    "original_url": str,         # 예: "https://www.youtube.com/watch?v=..."  → knowledge.original_url
    "metadata": {
        "video_title": str,      # → youtube_metadata.video_title  (+ knowledge.title 초기값)
        "channel_name": str,     # → youtube_metadata.channel_name
        "duration": int,         # → youtube_metadata.duration  (단위: ms — YouTubeService.get_metadata() 반환값 그대로)
    },
    "chunks": [
        {
            "chunk_order": int,  # → knowledge_chunk.chunk_order  (0부터)
            "content": str,      # → knowledge_chunk.content
            "start_time": int,   # → knowledge_chunk.start_time  (단위: ms)
        },
        ...
    ],
}
```

**동작 순서** (괄호 안은 ERD 기준 컬럼):
1. **`user_channel_identity` 조회** — `WHERE provider='kakao' AND provider_user_id = payload["kakao_user_id"]`
2. 없으면:
   - `user` 테이블에 신규 INSERT
     - `user_name`: NULL 또는 `'kakao_' + provider_user_id[:8]` (카카오는 닉네임 못 받음)
     - `is_active`: true
   - `user_channel_identity`에 매핑 INSERT `(user_id=위 user.id, provider='kakao', provider_user_id=payload["kakao_user_id"])`
3. `knowledge` INSERT
   - `user_id`: 위에서 얻은 user.id
   - `source_type`: `SourceType.YOUTUBE`
   - `title`: `metadata["video_title"]` (임시 제목; Step 2에서 AI 제목으로 UPDATE됨)
   - `original_url`: payload["original_url"]
   - `status`: `ProcessStatus.PROCESSING`
   - `category_id`: NULL (Step 2에서 채움)
   - `summary`: NULL (Step 2에서 채움)
   - → 생성된 `knowledge.id` (UUID) 확보
4. `youtube_metadata` INSERT (`knowledge_id`, `video_id`, `video_title`, `channel_name`, `duration`)
5. `knowledge_chunk` bulk INSERT
   - (`knowledge_id`, `content`, `start_time`, `chunk_order`)
   - `summary_detail`, `embedding`은 NULL로 두기 (Step 2에서 UPDATE)
6. 세션 commit
7. **생성된 `knowledge_id` (UUID) 반환**

**실패 시**: 세션 rollback 후 예외 재발생

---

### 1-2. `update_pipeline_result(data: dict) -> None`

**호출 시점**: Step 2 (LangGraph) 종료 직후, 채훈이 호출

**입력 data** (채훈이 pipeline 키 → ERD 컬럼 키 리네이밍 완료 후 전달):
```python
{
    "video_id": str,                  # → youtube_metadata.video_id 로 knowledge 역조회에 사용
    "metadata": {
        "video_title": str,           # → youtube_metadata.video_title
        "channel_name": str,          # → youtube_metadata.channel_name
        "duration": int,              # → youtube_metadata.duration  (ms)
    },
    "overview": {
        "title": str,                 # → knowledge.title (AI 생성 제목으로 UPDATE)
        "summary": str,               # → knowledge.summary (2차 요약본; 노션 업로드용)
        "category_name": str,         # ★ A안 — Repository가 category 테이블 upsert 후 knowledge.category_id로 세팅
        "source_type": "YOUTUBE",     # → knowledge.source_type
        "original_url": str,          # → knowledge.original_url
    },
    "chunks": [
        {
            "chunk_order": int,           # 매칭 키 (WHERE knowledge_chunk.chunk_order = ?)
            "content": str,               # → knowledge_chunk.content
            "start_time": int,            # → knowledge_chunk.start_time (ms)
            "summary_detail": str,        # → knowledge_chunk.summary_detail (1차 청크 요약)
            "embedding": list[float] | None,  # → knowledge_chunk.embedding (1536차원)
        },
        ...
    ],
}
```

**동작 순서**:
1. `video_id`로 `knowledge` 레코드 조회 (내부 UUID 확보)
2. **Category upsert** — `overview["category_name"]`으로 `category` 테이블 조회, 없으면 INSERT → `category_id` 확보
3. `knowledge` UPDATE
   - `title`, `summary`, `category_id`(위에서 얻은 값), `source_type`, `original_url`, `updated_at=now()`
   - ⚠️ **`status` 세팅은 제거해 주세요.** status는 `mark_completed`로 분리됩니다. (현재 코드에서 `status="COMPLETED"` 인라인 세팅된 부분 삭제)
4. `youtube_metadata` UPDATE (video_title, channel_name, duration)
5. `knowledge_chunk` UPDATE × N
   - WHERE (knowledge_id, chunk_order)
   - SET content, start_time, summary_detail, **embedding**
6. 세션 commit

**실패 시**: rollback 후 예외 재발생

**A안 핵심 — Category upsert 예시**:
```python
# overview["category_name"] 예: "요리", "운동", "재테크", 또는 LLM이 자유 생성한 새 이름
stmt = select(Category).where(Category.name == category_name)
existing = (await self.session.execute(stmt)).scalar_one_or_none()
if existing:
    category_id = existing.id
else:
    new_cat = Category(name=category_name)
    self.session.add(new_cat)
    await self.session.flush()   # id 확보
    category_id = new_cat.id
```

### 1-3. `mark_completed(knowledge_id: UUID) -> None` ★ 신규 추가

**호출 시점**: Step 3 (update_status)에서 채훈이 호출

**동작**:

```python
await self.session.execute(
    update(Knowledge)
    .where(Knowledge.id == knowledge_id)

await self.session.commit()
```

### 1-4. `mark_failed_by_video(video_id: str, kakao_user_id: str) -> None` ★ 신규 추가

**호출 시점**: 파이프라인 에러 발생 시 `handle_pipeline_failure`에서 채훈이 호출

**인자 주의** — `kakao_user_id`는 파이프라인 인자명일 뿐, ERD 컬럼이 아닙니다.
실제로는 `user_channel_identity.provider_user_id` (with `provider='kakao'`)에 해당합니다.

**왜 `video_id` + `kakao_user_id` 두 키가 필요한가?**
- 같은 유튜브 영상을 여러 유저가 저장할 수 있으므로 `video_id` 단독으로는 Knowledge를 특정할 수 없음
- Step 1 자체가 실패한 경우 `knowledge_id`가 생성되지 않았을 수 있으므로, on_error 콜백에 UUID를 직접 전달 불가

**동작 순서** (ERD 컬럼 명시):
1. `user_channel_identity`에서 `WHERE provider='kakao' AND provider_user_id = kakao_user_id` → `user_id` 조회
2. `youtube_metadata.video_id = video_id` AND `knowledge.user_id = user_id`로 Knowledge 조회
   - 없으면 조용히 종료 (Step 1 INSERT 전에 실패한 경우)
3. `knowledge.status = ProcessStatus.FAILED`로 UPDATE + commit

1. `alembic upgrade head`로 스키마 반영
2. Redis / Celery 워커 기동

   - `[Step 1] 레코드 생성 완료 (knowledge_id: ...)` — INSERT 성공
   - `[Step 2: LangGraph] 완료 — 카테고리: ...` — UPDATE 성공

   - `knowledge`, `youtube_metadata`, `knowledge_chunk`, `category`, `user`, `user_channel_identity` 전 테이블 행 존재 및 값 정상
